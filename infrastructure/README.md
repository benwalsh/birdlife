# infrastructure/ — the cloud mirror (culfinbirds.net)

Terraform for the **fan-out** target: the same Rails+React app the Pi runs, on
**ECS Express Mode** (Fargate + a managed ALB) behind **CloudFront**, mirroring the
Pi's detections into **RDS MySQL**. The Pi stays the source of truth; this is a lazy,
read-mostly copy.

```
 Pi ──push (direct to the ECS /ingest)──▶ ECS Express (Fargate) ──▶ RDS MySQL
                                                  ▲
 public ──▶ culfinbirds.net ──▶ CloudFront ───────┘   (cached, anonymous)
```

> **Runtime = ECS Express Mode**, AWS's successor to App Runner (which closed to new
> customers on 2026‑04‑30). Express Mode gives us a Fargate service, a shared ALB, and
> a `*.ecs.<region>.on.aws` endpoint from one module — CloudFront fronts it for the
> custom domain. See `ecs.tf`.

## What it creates

- **ECR** repo for the image (`Dockerfile` at the repo root builds it).
- **RDS MySQL** `db.t4g.micro`, private (only the ECS service can reach it), utf8mb4.
- **ECS Express Mode** service (`terraform-aws-modules/ecs//modules/express-service`)
  on an **ECS cluster**, running the image on Fargate with a managed ALB; secrets from
  SSM (execution role), RDS reachable via the `culfinbirds-ecs-tasks` security group.
- **CloudFront** + **ACM** (us-east-1) + **Route53** zone for the domain.
- **SSM SecureString** params — DB password, `SECRET_KEY_BASE`, ingest token (all
  generated by Terraform).

Requires the **AWS provider ≥ 6.34** (for `aws_ecs_express_gateway_service`) — see
`versions.tf`.

## Deploy

Two ordering rules, both learned the hard way:

1. **Delegate DNS first**, or ACM cert validation hangs (the cert is DNS-validated
   against the Route53 zone this creates; until the registrar points at that zone's
   nameservers, ACM can't see the records — `aws_acm_certificate_validation.cf` blocks,
   capped at 10m).
2. **Push the image to ECR before the full apply**, or the ECS service can't pull it
   and never goes healthy.

```bash
cd infra
terraform init   # (-upgrade if bumping the provider)

# 1. Create just the zone (for delegation) and ECR (to receive the image).
terraform apply -target=aws_route53_zone.main -target=aws_ecr_repository.app
terraform output route53_nameservers

# 2a. At the registrar / DNS host (culfinbirds.net is on DNSimple), set those four
#     nameservers, then wait for propagation:
dig +short NS culfinbirds.net        # should return the awsdns.* servers

# 2b. Build + push the image — MUST be linux/amd64 (Fargate is x86_64):
REPO=$(terraform -chdir=infra output -raw ecr_repository_url)
[ -n "$REPO" ] || { echo "REPO empty — run step 1 first"; }
aws ecr get-login-password --region eu-west-1 | docker login --username AWS --password-stdin "$REPO"
docker build --platform linux/amd64 -t "$REPO:latest" ..   # from repo root
docker push "$REPO:latest"

# 3. Full apply — cert validates (~1-2m) and the ECS service pulls the image + boots.
terraform apply

# On the Pi's .env (values from Terraform outputs):
#   CLOUD_INGEST_URL   = $(terraform output -raw ingest_url)
#   CLOUD_INGEST_TOKEN = $(terraform output -raw cloud_ingest_token)
```

## Alert emails (SES)

`ses.tf` sets up the alert path: a DKIM-verified domain identity, a custom MAIL FROM
(SPF), a DMARC record, and `ses:SendEmail` on the ECS task role. No email template
lives in Terraform — the Rails `Notifier` builds both the alert and the digest as SES
*simple* content (HTML/text in Ruby). `ALERTS_FROM` on the ECS service switches sending
on (fail-soft — sends that fail just retry on the next ingest tick).

**Two manual steps `terraform apply` can't do:**
1. **Wait for DKIM verification.** After apply, SES verifies the domain via the DKIM
   CNAMEs (minutes to a few hours). Check: SES console → Identities → `culfinbirds.net`.
2. **Request production access.** SES starts in the **sandbox** — it can only email
   *verified* addresses. Real sends to subscribers need production access: SES console →
   Account dashboard → **Request production access** (a support ticket, ~1–2 days). Do
   this early. To test meanwhile, verify your own address as an identity and email yourself.

## Notes / still to tune

- **Migrations** run on container boot via `bin/docker-entrypoint` (`db:prepare` is
  idempotent and waits for the DB; with a single warm task there's no race). RDS is
  private, so there's no local `rails db:migrate` path.
- **New image / redeploy**: push a new `:latest` to ECR, then force a new ECS
  deployment (Express Mode doesn't auto-deploy on push the way App Runner did). The
  GitHub Actions workflow (task #20) will automate build → push → force-deploy.
- **Assets**: the image serves its own static files (incl. the ~225 MB of
  illustrations) and CloudFront caches them. To slim the image later, offload the
  illustrations to S3 and set `RAILS_SERVE_STATIC_FILES=false` + `ASSET_HOST`.
- **Auth on the public mirror**: CloudFront strips cookies for cache hits, so the
  cloud site is effectively anonymous. Admin stays on the Pi (via Tailscale).
- **Remote state**: wire the S3 backend in `versions.tf` before real use.
- Rough cost: ~$45–65/mo (RDS + the Fargate task + the ALB dominate; Express Mode
  itself has no surcharge).
